# Supabase Setup Checklist for Greetflow

## A. Data Type Compatibility

### ✅ Compatible Data Types
- **UUID**: `uuid_generate_v4()` - Fully supported
- **text**: Preferred over `varchar` for Supabase
- **timestamp with time zone**: Recommended for all date/time fields
- **jsonb**: Fully supported with indexing
- **boolean**: Standard PostgreSQL type
- **text[]**: Array types supported
- **date**: Standard PostgreSQL type

### ⚠️ Data Types Requiring Attention
- **SERIAL/IDENTITY**: Use `GENERATED BY DEFAULT AS IDENTITY` instead of `SERIAL`
- **JSON vs JSONB**: Always use `jsonb` for better performance and indexing
- **VARCHAR vs TEXT**: Use `text` for better Supabase compatibility

### 🔄 Conversion Examples
```sql
-- OLD (avoid)
id SERIAL PRIMARY KEY,
description VARCHAR(255)

-- NEW (recommended)
id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
description text
```

## B. Auto-Increment / Primary Keys

### Recommended Approach
Supabase works best with UUID primary keys for distributed systems and real-time features.

```sql
-- Recommended: UUID with uuid_generate_v4()
CREATE TABLE people (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    email text NOT NULL UNIQUE
);

-- Alternative: Integer with identity (if needed)
CREATE TABLE sequences (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL
);
```

### Benefits of UUID in Supabase
- Better for real-time subscriptions
- No conflicts in distributed environments
- Works seamlessly with Supabase Auth
- Enables offline-first applications

## C. Reserved Keywords

### PostgreSQL Keywords to Avoid
Avoid these reserved words for table/column names:

```sql

view, function, trigger, function, procedure, trigger, 
constraint, foreign, primary, unique, check, default, 
null, not, and, or, in, exists, between, like, is, 
case, when, then, else, end, if, while, for, loop, 
return, begin, commit, rollback, savepoint, transaction

-- ✅ USE these alternatives
users, orders, groups, selected, source, tables, 
indexes, views, functions, procedures, triggers, 
constraints, foreign_keys, primary_keys, unique_keys, 
checks, defaults, nulls, nots, ands, ors, ins, 
exists_in, between_values, likes, is_null, cases, 
whens, thens, elses, ends, ifs, whiles, fors, loops, 
returns, begins, commits, rollbacks, savepoints, transactions
```

## D. Row Level Security (RLS)

### Enable RLS on Tables
```sql
-- Enable RLS on all user-facing tables
ALTER TABLE people ENABLE ROW LEVEL SECURITY;
ALTER TABLE campaigns ENABLE ROW LEVEL SECURITY;
ALTER TABLE sends ENABLE ROW LEVEL SECURITY;
ALTER TABLE audit_logs ENABLE ROW LEVEL SECURITY;

-- Keep system tables without RLS (optional)
-- oauth_tokens, templates can remain without RLS for system access
```

### Sample RLS Policies

#### 1. Users can access only their own data
```sql
-- Policy for people table - users see only their own records
CREATE POLICY "Users can view own people" ON people
    FOR SELECT USING (auth.uid()::text = created_by::text);

CREATE POLICY "Users can update own people" ON people
    FOR UPDATE USING (auth.uid()::text = created_by::text);

CREATE POLICY "Users can insert own people" ON people
    FOR INSERT WITH CHECK (auth.uid()::text = created_by::text);
```

#### 2. Admins have full access
```sql
-- Admin role check (assuming you have a users table with roles)
CREATE POLICY "Admins have full access to people" ON people
    FOR ALL USING (
        EXISTS (
            SELECT 1 FROM auth.users 
            WHERE auth.users.id = auth.uid() 
            AND auth.users.raw_user_meta_data->>'role' = 'admin'
        )
    );
```

#### 3. Campaign access based on ownership
```sql
-- Campaigns: users see campaigns they created
CREATE POLICY "Users can view own campaigns" ON campaigns
    FOR SELECT USING (auth.uid()::text = created_by::text);

-- Sends: users see sends for their campaigns
CREATE POLICY "Users can view related sends" ON sends
    FOR SELECT USING (
        EXISTS (
            SELECT 1 FROM campaigns 
            WHERE campaigns.id = sends.campaign_id 
            AND campaigns.created_by = auth.uid()::text
        )
    );
```

## E. Environment Variables / Secrets

### Supabase Secrets Management
Store sensitive data using Supabase Secrets:

```sql
-- Store SMTP credentials
INSERT INTO vault.secrets (name, secret) VALUES 
('smtp_host', 'smtp.gmail.com'),
('smtp_port', '587'),
('smtp_username', 'your-email@gmail.com'),
('smtp_password', 'your-app-password');

-- Store OAuth credentials
INSERT INTO vault.secrets (name, secret) VALUES 
('gmail_client_id', 'your-google-client-id'),
('gmail_client_secret', 'your-google-client-secret');
```

### Accessing Secrets in Edge Functions
```typescript
// In Supabase Edge Functions
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'

const supabase = createClient(
  Deno.env.get('SUPABASE_URL')!,
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
)

// Access secrets
const { data: smtpHost } = await supabase.rpc('get_secret', { name: 'smtp_host' })
```

## F. Recommended Best Practices

### 1. Indexing Strategy
```sql
-- Composite indexes for common query patterns
CREATE INDEX CONCURRENTLY idx_sends_status_scheduled_person 
ON sends(status, scheduled_for, person_id);

-- Partial indexes for active records
CREATE INDEX CONCURRENTLY idx_active_templates 
ON templates(id) WHERE is_active = true;

-- GIN indexes for JSONB queries
CREATE INDEX CONCURRENTLY idx_campaigns_audience_gin 
ON campaigns USING GIN (audience_query);
```

### 2. Backup and Migration Strategy
```sql
-- Create migration tracking table
CREATE TABLE IF NOT EXISTS schema_migrations (
    version text PRIMARY KEY,
    applied_at timestamp with time zone DEFAULT NOW()
);

-- Track applied migrations
INSERT INTO schema_migrations (version) VALUES ('001_initial_schema');
INSERT INTO schema_migrations (version) VALUES ('002_add_indexes');
```

### 3. Testing Data Insertion Safely
```sql
-- Use transactions for safe testing
BEGIN;

-- Test insert with rollback capability
INSERT INTO people (first_name, last_name, email, consent_email) 
VALUES ('Test', 'User', 'test@example.com', true);

-- Verify the insert
SELECT * FROM people WHERE email = 'test@example.com';

-- Rollback if issues found
ROLLBACK; -- or COMMIT; if successful
```

### 4. Schema Migration Best Practices
```sql
-- Always use IF NOT EXISTS for safety
CREATE TABLE IF NOT EXISTS new_table (...);

-- Add columns with defaults
ALTER TABLE people ADD COLUMN IF NOT EXISTS phone text;

-- Create indexes concurrently to avoid locks
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_people_phone 
ON people(phone) WHERE phone IS NOT NULL;
```

### 5. Performance Monitoring
```sql
-- Enable query statistics
ALTER SYSTEM SET track_activities = on;
ALTER SYSTEM SET track_counts = on;
ALTER SYSTEM SET track_io_timing = on;

-- Monitor slow queries
SELECT query, mean_time, calls 
FROM pg_stat_statements 
ORDER BY mean_time DESC 
LIMIT 10;
```
